---
title: API Stability
description: Compatibility rules for Chatto's protobuf API surfaces.
---

Chatto is pre-1.0, but its protobuf API is still a compatibility contract.
Breaking changes are allowed only when they are intentional and documented in
the pull request.

## Stability Tiers

| Tier | Surface | Promise |
| --- | --- | --- |
| Integration API | `chatto.api.v1` ConnectRPC services under `/api/connect` | Prefer additive evolution. Field numbers, field types, service names, method names, enum numbers, pagination semantics, and common Connect error codes are caller-visible API. |
| Bundled app API | Exceptional app-specific protobuf APIs, if added later | May be less SDK-oriented, but must tolerate reasonable bundled client/server version skew. |
| Realtime protocol | `chatto.api.v1.Realtime*` frames at `/api/realtime` | Add frames and fields compatibly where possible. New required client behavior must be negotiated or versioned. |

`chatto.api.v1` is the default home for integration and frontend-used APIs. An
app-specific namespace should be rare and should exist only when the behavior is
inherently tied to the bundled web client.

## ConnectRPC Rules

- Do not renumber fields.
- Do not change a field type at an existing tag.
- Reserve both a removed field tag and name.
- Prefer adding fields, enum values, methods, and services over changing
  existing ones.
- Treat renames as source-breaking even when the protobuf wire remains
  compatible.
- Use `PageRequest` and `PageInfo` for offset-list APIs.
- Use opaque cursors for timeline/window APIs.
- Return `NOT_FOUND` for absent singular resources unless the response
  explicitly documents absence as a successful nullable result.
- Keep common Connect error codes stable for authentication, authorization,
  validation, not-found, conflict, and precondition outcomes.

## Realtime Rules

The realtime WebSocket protocol is live-only in version 1. Clients recover
missed state by issuing projected ConnectRPC reads after reconnecting.

Future realtime changes should be additive where possible. A server that
requires new client behavior must advertise that behavior in hello/capability
fields or introduce a new protocol version.

## Generated Reference

The `.proto` files in `proto/chatto/api/v1/` are the source of truth. Generated
reference pages are split into ConnectRPC service groups, shared types and
enums, and realtime protocol frames.
